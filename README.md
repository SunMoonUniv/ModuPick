# 모두픽(ModuPick): 유쾌하고 공정한 팀 의사결정 도구

## **1. 제품 개요**

### **1.1 배경 및 문제점**

팀 프로젝트(조별 과제, 스터디, 사내 TF 등) 초기 단계에서 역할 분담, 팀장 선정 시 발생하는 '결정 장애'와 '눈치 싸움'은 불필요한 감정 소모와 시간 낭비를 초래함. 기존의 오프라인 사다리 타기나 단순 투표 툴은 실시간 몰입감이나 재미 요소가 부족함.

### **1.2 해결책**

웹소켓 기반의 실시간 멀티플레이어 환경에서 유쾌한 미니게임을 통해 의사결정을 내릴 수 있는 웹 플랫폼 제공. 책임을 '운'과 '익명성'에 돌려 감정 소모를 없애고 빠른 결론을 유도함.

## **2. 타겟 고객**

- **Primary:** 20대 대학생 (조별 과제, IT 동아리) 및 스터디 그룹
- **Secondary:** 2030 직장인 (사내 TF, 가벼운 팀 빌딩)

## **3. 성공 지표**

MVP 런칭 후 다음 지표를 추적하여 제품의 성공 여부를 판단합니다.

- **Acquisition (획득):** 방 생성 대비 초대 코드를 통한 신규 유저 유입 비율 (목표: 1방당 평균 3명 이상 유입)
- **Activation (활성화):** 대기방 입장 후 실제 게임 시작(완료)까지 이어지는 세션 전환율 (목표: 80% 이상)
- **Retention/Referral (유지/추천):** 게임 종료 후 '결과 이미지 공유' 버튼 클릭률 및 재방문 방 생성 비율

## **4. 기능 요구사항**

### **4.1 대기방 및 유저 플로우**

| **기능 명칭** | **상세 요구사항** | **우선순위** |
| --- | --- | --- |
| **방 생성** | 방장은 방 제목과 최대 인원(최대 30명) 설정 후 방 생성. 서버는 6~8자리의 무작위 영문+숫자 초대 코드를 발급함. | P1 |
| **방 입장** | 로그인 기능 없음. 유저는 메인 페이지에서 초대 코드를 입력하고, 닉네임/프로필/소개(선택)를 입력한 뒤 입장함. | P1 |
| **실시간 대기방** | 웹소켓 연결을 통해 입장/퇴장 이벤트, 채팅 텍스트, 팀원의 'Ready' 상태가 실시간으로 브로드캐스트됨. | P1 |
| **게임 주도권** | 방장(Host)에게만 '게임 선택' 및 '시작' 트리거 버튼이 활성화됨. 팀원은 '준비 완료' 버튼만 조작 가능. | P1 |

### **4.2 실시간 미니게임 (Mini Games)**

모든 게임은 서버와의 실시간 통신을 통해 모든 클라이언트 화면에서 지연 없이 동일한 결과를 연출해야 합니다.

| **게임명** | **게임 룰 및 요구사항** | **우선순위** |
| --- | --- | --- |
| **[Game 1] 룰렛** | 팀원 캐릭터가 배치된 룰렛이 방장 조작 시 전원 화면에서 동기화되어 회전. | P1 |
| **[Game 2] 사다리** | 상단에 참가자, 하단에 역할(N개) 배치. 애니메이션이 순차/동시 진행되어 역할 분배. | P1 |
| **[Game 3] 킹메이커** | 익명 텍스트(팀명 등) 입력 ➔ 수집 ➔ 서바이벌 투표 또는 룰렛으로 최종 1개 선정. | P2 |
| **[Game 4] 시간초 잡기** | 목표 시간(예: 5.00초) 설정. 각 클라이언트에서 정지 클릭. (클라이언트 타임스탬프와 서버 타임스탬프 오차 보정 로직 필요). 오차가 가장 큰 유저가 당첨(벌칙). | P1 |
| **[Game 5] 익명 저격** | 방장이 프리셋 질문 선택 ➔ 10초 내 타겟 유저 익명 탭(클릭). ➔ 결과 공개 시 레이저 애니메이션 연출. 최다 득표자 동점 시 서든데스 룰렛 자동 실행. | P1 |
| **[Game 6] 눈치게임** | [UP] 버튼 클릭. 서버 기준 0.3초 내 중복 입력 시 해당 인원 탈락. 최후 1인 탈락. 시간 경과 시 시각적 압박(붉은 화면 점멸) 연출 추가. | P2 |

### **4.3 결과 및 공유**

- **결과 화면:** 당첨자(또는 분배 결과)가 화면에 크게 표시됨.
- **이미지 내보내기:** HTML/DOM 영역을 이미지(png/jpg)로 렌더링하여 클립보드 복사 또는 기기 저장 기능 제공 .

## **5. 비기능 요구사항**

### **5.1 UI/UX**

- **시각적 특징:** 굵은 흑백 테두리, 원색 형광 컬러(Neon Yellow/Blue), 거대한 타이포그래피. '오락실' 같은 타격감.
- **애니메이션:** 무거운 3D나 긴 트랜지션 금지. 즉각적이고 짧은 2D 컷아웃 모션, 디바이스 진동(Haptic) API 적극 활용.
- **UX 주의점:** 접속 후 2-depth 이내에 대기방에 입장할 수 있도록 UI 간소화.

### **5.2 기술 스택 및 인프라**

- **Frontend:** React.js, Tailwind CSS, Framer Motion (or HTML5 Canvas)
- **Backend:** Python FastAPI (WebSocket 처리)
- **DB:** PostgreSQL 또는 MySQL
- **Infra/Architecture:** Docker Compose 기반 컨테이너화. (Phase 1 이후 k8s 오케스트레이션 및 Redis Pub/Sub 도입)
- **성능 SLA:** 웹소켓 이벤트 핑/폰(Ping-Pong) 지연 시간은 **최대 100ms 이하**를 유지해야 눈치게임 등의 플레이 경험이 훼손되지 않음.

## **6. 제외 범위**

본 PRD(Phase 1~2)에서는 다음 기능을 제외하며, 서비스 안정화 후 Phase 3에서 도입합니다.

- 회원가입 및 소셜 로그인 (OAuth)
- Slack / Discord Webhook 및 App 연동 (명령어 생성 등)
- 사용자 커스텀 아바타 (기본 제공 프로필 사진으로 대체)

## **7. 검증 및 리스크**

- **UX 리스크 (초대 코드 입력):** 모바일 환경에서 전달받은 코드를 웹 메인 화면에 직접 타이핑하는 과정에서 유저 이탈(Drop-off)이 발생할 확률이 매우 높습니다. MVP 런칭 직후 **'초대 코드 입력 뷰'의 체류 시간과 이탈률을 모니터링**해야 하며, 지표가 저조할 경우 URL 파라미터를 통한 '대기방 자동 진입(Zero Onboarding)' 방식으로 즉각 선회할 수 있도록 프론트엔드 라우팅 구조를 유연하게 설계해야 합니다.
- **동기화 리스크 (네트워크 지연):** [Game 4 시간초 잡기]와 [Game 6 눈치게임]은 클라이언트 기기의 네트워크 상태에 따라 불공정성 이슈가 제기될 수 있습니다. 이벤트 판정 기준을 철저히 '서버 타임스탬프 도착' 기준으로 통일하고, 0.3초의 관대한 동시 입력 판정 버퍼를 두어 해결합니다.
